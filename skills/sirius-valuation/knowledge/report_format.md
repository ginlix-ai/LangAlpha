# 分析报告格式与内联引用规范

> 本文档定义 Sirius 七维度分析中每篇维度报告的结构与引用规范。
> 报告是分析的**第一产出物**（底稿），结构化 json 是从报告中**提取**的。

---

## 报告目的

1. **留分析底**：完整记录判断逻辑，支持后续迭代（修改 D7 参数后能追溯原始分析）
2. **内联溯源**：每个观点可追踪到原始数据/计算/外部信息
3. **前端联动**：结构化数据中的每个字段引用报告章节，用户点击可跳转阅读完整分析

---

## 报告结构模板

```markdown
# D{N} {维度名称} — {公司名称}

> 分析日期：{date} | 数据截止：{latest_fiscal_year}

## 1. {分析模块标题}

{分析正文，每个关键判断带脚注}[^ref_d{N}_01]

### 关键指标

| 指标 | 数值 | 来源 |
|------|------|------|
| xxx_growth_pct | 14.6% | [^ref_d{N}_02] |

## 2. {下一个分析模块}

...

## {M}. 综合评分与逻辑链

**评分：{score}/10**（置信度：{confidence}）

**逻辑链**：{因素A}(+N) → {因素B}(+N) → {因素C}(-N) = {score}

评分依据[^ref_d{N}_10][^ref_d{N}_11]：...

## {M+1}. 核心发现

1. {finding_1}[^ref_d{N}_XX]
2. {finding_2}[^ref_d{N}_XX]

## {M+2}. 风险提示

- {risk_1}[^ref_d{N}_XX]
- {risk_2}[^ref_d{N}_XX]

## References

[^ref_d{N}_01]: **{标题}** | `{source}` | {摘要}
[^ref_d{N}_02]: **{标题}** | `calculation` | {计算公式}
...
```

---

## 脚注定义格式

```
[^ref_id]: **标题** | `source_spec` | 摘要说明（1-3句话）
```

### source_spec 格式

| 类型 | 格式 | 示例 |
|------|------|------|
| 数据文件 | `raw/{filename}` | `raw/incomeStatement.json` |
| 引擎结果 | `engine_result.json` | |
| 计算推导 | `calculation` | 后面的摘要写逐步公式 |
| 外部链接 | `external:{url}` | `external:https://hkexnews.hk/...` |
| 知识文档 | `knowledge/{file}#{anchor}` | `knowledge/d3_environment.md#regulatory` |
| 用户备忘 | `memo:{key}` | `memo://user_research_notes` |

### 脚注 ID 规则

- 格式：`ref_d{N}_{seq}`（如 `ref_d3_01`, `ref_d3_02`）
- 每个维度独立编号，从 01 开始
- 同一个来源在不同地方引用用同一个 ref_id

---

## 章节标题作为引用锚点

前端通过 H2 标题生成锚点。报告中的 H2 标题要求：
- 有意义的分析主题（不是"第一部分"这种无意义标题）
- 稳定不变（结构化提取时用标题作为 `section` 字段值）

**示例**：
```markdown
## 2. 行业增长驱动力是否在发生结构性变化
```
→ 锚点：`行业增长驱动力是否在发生结构性变化`
→ 结构化 json 中引用：`"section": "行业增长驱动力是否在发生结构性变化"`

---

## 与结构化 JSON 的映射关系

结构化 json 中的 `references` 数组从报告脚注转换而来，新增 `section` 字段：

```json
{
  "id": "ref_d3_02",
  "type": "external",
  "title": "管理层电话会议",
  "source": "https://...",
  "detail": "CEO 在 Q4 电话会提到 AI 投入翻倍",
  "section": "行业增长驱动力是否在发生结构性变化",
  "cited_by": ["key_findings[1]", "metrics.structural_change"]
}
```

**`section` 字段**：
- 值为报告中引用该脚注的 H2 标题文本（去掉序号前缀"N. "）
- 前端用这个字段定位文件中的章节位置
- 一个 ref 被多处引用时，取第一次出现的章节

---

## 报告质量要求

### 内容深度
- 不是简单罗列数据，要有**判断逻辑**的完整推导
- 每个结论要说"因为 X 数据显示 Y，所以我判断 Z"
- 如果涉及计算，要写出完整公式和中间步骤

### 引用密度
| 报告内容 | 最低引用 |
|---------|---------|
| 评分段落 | ≥ 2 个脚注 |
| 每个判断/结论 | ≥ 1 个脚注 |
| 量化指标 | ≥ 1 个脚注 |
| 风险判断 | ≥ 1 个脚注 |
| 整篇报告 | ≥ 5 个脚注 |

### 报告长度
- 每篇报告 800-2000 字（不含 References）
- D6（综合评估）允许更长（依赖所有 D1-D5）
- References 区域不限长度

---

## 结构化提取指引

Phase 3 提取时，读取每篇报告，按 `output_schema.md` 定义的参数逐字段提取：

1. **score** → 从"综合评分与逻辑链"章节提取分数
2. **metrics** → 从"关键指标"表格和正文加粗数值提取
3. **key_findings** → 从"核心发现"列表提取
4. **risks** → 从"风险提示"列表提取
5. **summary** → 从报告开头或评分段落提炼 1-2 句概括
6. **references** → 从 References 区域解析脚注，附加 `section` 和 `cited_by`
7. **confidence** → 从评分段落提取置信度

提取后的 json 写入 `data/{symbol}/structured/d{N}.json`。
